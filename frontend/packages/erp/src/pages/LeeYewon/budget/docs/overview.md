# 버핏그라운드 지점 예산 관리 시스템

> 12개 버핏그라운드 지점의 월/분기/연간 예산을 추적하고, 지출을 등록·검증·분석하는 웹 기반 시스템.

---

## 1. 한눈에 보기

- **프로젝트명**: `branch-operations` (상위) / `budget` (첫 모듈)
- **목적**: 시트 기반 예산/지출 관리를 시스템으로 이관
- **사용자**: 지점 직원 (12개 지점) + 본사 관리자 (PM, SGM, GM, 운영기획팀 팀장)
- **오픈 시점**: 2026년 5월 (가정)
- **기술 스택**: FastAPI + React + SQLite(개발)/PostgreSQL(프로덕션)
- **개발 방식**: 1인 바이브 코딩 (Claude Code + 역할별 서브 에이전트)
- **예상 기간**: 순수 개발 2주 / 본업 병행 1개월

## 2. 왜 만드는가

### 2-1. 현재 상태

12개 버핏그라운드 지점은 각자 **Google Sheets**로 예산을 관리한다. 신도림 지점 기준:

- **대시보드 시트**: 6블록 (경영계획 예산 VAT-/VAT+, 월별 실지출 VAT-/VAT+, 월별 잔여, 분기별 잔여)
- **지출관리 시트**: 13개 필드 (구분/주문일자/귀속연월/수령확인/작성자/계정/항목/단가/수량/배송비/총액/비고/링크)
- 2026년 1~4월 기준 지출 **241건**, 주요 작성자 박영준 (193건)

시트 운영은 다음과 같이 돌아간다:
1. 지점 직원이 지출 발생 시 지출관리 시트에 직접 입력
2. 대시보드 시트는 수식으로 자동 집계 (월/분기 잔여 예산)
3. 분기 초과 시 Flex 결재로 추경 처리 → 완료 후 시트에 메모
4. 본사는 개별 지점 시트를 일일이 열어봐야 전사 현황 파악

### 2-2. 문제점

**데이터 품질**
- **카테고리 미입력 12건** (140만원) 방치 — "가이드 범주 밖" 품목(에어컨 필터, 제빙기 부품 등)
- 반복 입력 패턴 불투명 (샴푸 68회, 하루에 5~7회 같은 품목)
- 36건 "링크 없음" — 오프라인 구매 vs 기록 누락 구분 어려움
- 수령 확인 누락 5일 이상 2건

**운영 효율**
- 자주 쓰는 품목도 매번 풀 입력 (자동완성 없음)
- 지점 간 데이터 비교 어려움 (12개 시트 비교)
- 분기 초과 임박 감지가 수동 (매일 시트 봐야 함)

**집계 신뢰도**
- 시트 수식 오류 가능성, 권한 사고(행 실수 삭제) 복구 어려움
- 과거 값 추적 불완전 (변경 이력 없음)

**본사 관점 가시성**
- 12개 지점 실시간 통합 대시보드 없음
- 지점별 소진 패턴 비교, 공통 이슈 식별 어려움
- 추경 발생 빈도/규모 분석 어려움

### 2-3. 시스템으로 얻으려는 것

**Phase 1 (MVP) 목표**

1. **시트 1:1 이관**: 기존 관행을 구조 변경 없이 시스템으로 이관
2. **데이터 정합성**: 변경 이력, 감사 로그, Soft delete로 복구 가능성 확보
3. **본사 통합 뷰**: 12개 지점 현황을 한 화면에
4. **예산 경고 자동화**: 90%/100% 임계값 자동 감지
5. **입력 효율**: 품목 자동완성, 중복 감지로 반복 작업 줄이기
6. **미정 카테고리 플로우**: 애매한 품목도 막힘 없이 등록 가능

**Phase 2 이후**

- 슬랙 알림 연동
- 영수증 이미지 업로드
- 운영 설정 UI (경고 임계값 조정)
- 재입력 편의 기능 (지난 달 복사)

## 3. 주요 사용자와 역할

### 3-1. 역할

| 역할 | 설명 | 주 사용 화면 |
|---|---|---|
| **지점 직원** (branch_staff) | 12개 지점의 운영 담당자. 지출 등록/조회/수정 | 지점 뷰 (월별 대시보드, 지출 등록) |
| **본사 PM** (hq_pm) | 전체 지점 현황 모니터링 | 본사 뷰 (조회 전담) |
| **본사 SGM** (hq_sgm) | 운영 개입, 카테고리 재분류, 타 지점 수정 | 본사 뷰 + 미정 재분류 |
| **본사 GM** (hq_gm) | 예산/추경/마스터 전권 | 본사 뷰 + 추경 입력 + 마스터 관리 |
| **운영기획팀 팀장** (hq_planning_lead) | 카테고리 전용 승인 | 본사 뷰 + 전용 승인 |

### 3-2. 주요 사용 시나리오

**지점 직원 (매일)**
- 쿠팡에서 샴푸 주문 → 시스템에 지출 등록 → 수령 후 수령확인 체크
- 월 중 카테고리 소진율 확인 → 90% 도달 시 주의 배지 확인
- 카테고리 불분명한 품목은 "미정"으로 등록, 사유 입력

**본사 PM (매일)**
- 본사 뷰에서 12개 지점 현황 한눈에
- 분기 90% 임박 지점 식별
- 미정 대기 목록 확인 (SGM 재분류 필요)

**본사 SGM (필요 시)**
- 미정 카테고리 재분류
- 특이 지출 수정/삭제 (본인 지점 아닌 경우)

**본사 GM (월 1~2회)**
- Flex에서 추경 결재 완료 → 시스템에 추경 입력
- 원 예산 수정 (드뭄)
- 연초 연간 예산 CSV 업로드

**운영기획팀 팀장 (분기 1~2회)**
- 지점 요청 카테고리 전용 승인

## 4. 시스템 범위

### 4-1. 포함 (Phase 1 MVP)

**기능**
- 지출 등록/조회/수정/삭제 (소프트)
- 환불 처리 (부분/전액)
- 수령 확인 토글
- 미정 카테고리 플로우 (등록 → 알림 → 재분류)
- 지점 뷰 월별/연간 대시보드
- 본사 뷰 통합 대시보드 (KPI, 히트맵, 스택 막대)
- 추경 입력 (GM)
- 카테고리 전용 (요청 → 승인 → 실행)
- 원 예산 수정 (GM)
- 연간 예산 CSV 업로드
- 품목 마스터 자동 생성/자동완성
- 중복 감지 (3회 이상 경고)
- 감사 로그 (모든 변경)
- 1~4월 기존 데이터 이관

**역할 기반 권한**
- 5개 역할, Permission 매트릭스
- 지점 직원 자기 지점 CRUD + 전체 지점 조회
- 본사 역할별 세분화

### 4-2. 제외 (Phase 2 이후)

- 슬랙 알림 연동 (MVP는 시스템 내 알림 센터만)
- 영수증 이미지 업로드 (MVP는 URL 링크만)
- 고정비 자동 생성 (영구 수기 결정)
- 운영 설정 UI (MVP는 코드 하드코딩)
- 연말 락 기능 (`is_locked`)
- 다음 연도 이월 로직
- HR 시스템 연동 (인건비 자동화)
- 모바일 전용 앱 (웹 반응형으로 커버)

### 4-3. 명시적 비범위

- 지점 운영관리 통합 프로젝트의 다른 2개 축 (체크리스트 / 환경 모니터링)은 별도 모듈
- 판매/매출 관리 (bstore-analytics 별도 시스템)
- 발주/입고 관리 (bstore-order 별도 시스템)
- 회계 연동 (Flex 외부 처리)

## 5. 아키텍처 개요

### 5-1. 구성

```
┌─────────────────────────────────────────────┐
│  React Frontend (Vite + Tailwind)           │
│  - 지점 뷰 (월별/연간 대시보드, 지출 CRUD)     │
│  - 본사 뷰 (통합 대시보드, 추경 입력)          │
└────────────────┬────────────────────────────┘
                 │ REST API (JSON)
┌────────────────▼────────────────────────────┐
│  FastAPI Backend                            │
│  - Auth (JWT)                               │
│  - Permission middleware                    │
│  - Audit log middleware                     │
│  - Business rules engine                    │
└────────────────┬────────────────────────────┘
                 │
┌────────────────▼────────────────────────────┐
│  Database                                   │
│  - SQLite (개발) / PostgreSQL (프로덕션)       │
│  - 13 tables (data-model.md 참조)            │
└─────────────────────────────────────────────┘
```

### 5-2. 핵심 모듈

**Frontend**
- Dashboard 컴포넌트 (지점/본사 뷰)
- ExpenseForm (등록/수정 공통)
- ExpenseList (필터, 정렬, 배지)
- BudgetMatrix (연간 매트릭스)
- Heatmap (본사 뷰)
- Modal 컴포넌트 (환불, 미정 재분류, 추경 입력)

**Backend**
- `auth/` — JWT 기반 인증, 세션 관리
- `permissions/` — 역할 × Permission 매트릭스
- `budget/` — 예산 집계, 실질 예산 계산
- `expenses/` — 지출 CRUD + 환불 + 재분류
- `adjustments/` — 추경 + 카테고리 전용
- `catalog/` — 품목 마스터 자동 관리
- `audit/` — 감사 로그 기록/조회
- `migration/` — 1~4월 이관 스크립트
- `notifications/` — 알림 생성 (Phase 1 DB only)

**Database**
- 13 tables (상세: `data-model.md`)
- 감사 로그 JSONB
- Soft delete 기본

### 5-3. 통합 지점

**외부 시스템 참조**

- **Flex (결재 플랫폼)**: 추경 결재는 Flex에서 진행. 시스템은 Flex 결재번호(`flex_approval_ref`)만 저장.
- **Slack (Phase 2)**: 주요 알림 슬랙 DM/채널 연동
- **Google Drive (선택)**: 영수증 이미지 저장 검토 (Phase 2)

**다른 내부 시스템과의 관계**

- **bstore-analytics**: 판매 데이터 분석, 별개 시스템
- **bstore-order**: 발주/입고 관리, 별개 시스템
- **이 시스템(budget)**: 지출 관리, 독립적

향후 공통 지점(인증, 유저 마스터)은 통합 고려 가능하지만 MVP는 독립.

## 6. 핵심 개념 빠른 설명

**카테고리 체계**
- 대카테고리 7개 (경상 소모품 / 비경상 소모품 / 기타 비용 / 세탁 / 미화 / 파트 인건비 / **미정**)
- 소카테고리 11개 (대카테고리별 1~N개)
- "미정"은 임시 보관함, 집계에서 제외

**VAT 처리**
- 저장은 VAT+ 기준
- 표시는 VAT+ 기본, 토글로 VAT- 전환 가능
- `VAT- = round(VAT+ / 1.1)`

**예산 단위**
- 원 예산: `annual_budgets` (지점 × 소카테고리 × 연-월)
- 추경: `budget_adjustments` (Flex 결재 완료 후 GM 입력)
- 카테고리 전용: `budget_adjustments` (transfer_out + transfer_in 쌍)
- **실질 예산 = 원 예산 + 승인된 추경 누적 + 전용 증가분**

**예산 경고**
- 정상: < 90%
- 주의: 90% ≤ x < 100% (노란 배지)
- 위험: ≥ 100% (빨간 배지)
- 월/분기/연간 공통 기준

**지출 상태**
- `completed` (정상)
- `partially_refunded` (부분 환불)
- `fully_refunded` (전액 환불, 취소 포함)

**수령 지연 플래그**
- 주문일 + 7일 경과 + `receipt_confirmed = FALSE` → "주의" 배지
- `is_long_delivery = TRUE`면 14일 기준

**권한 원칙**
- 지점 직원: 자기 지점 CRUD + 전체 지점 조회
- 본사: 조회는 전체, 수정 권한 역할별 차등
- 모든 변경은 `audit_logs`에 기록

**감사 로그**
- 생성/수정/삭제/환불/승인 모든 액션
- before/after JSONB 스냅샷
- Soft delete 기본

## 9. 제약과 가정

### 9-1. 기술 제약

- **1인 바이브 코딩**: 에이전트(backend/frontend/ui-designer/planner) 활용해도 결국 검수/검증은 1인. MVP 범위 보수적.
- **데이터 스키마 안정성**: Phase 2~3에서 큰 스키마 변경 최소화. 현 스키마 설계가 중요.
- **배포 환경**: 버핏서울 내부 서버 또는 Vercel/Render 등 PaaS. Phase 0에서 결정.

### 9-2. 운영 가정

- **지점 직원 PC 사용**: 데스크톱/노트북 기준 설계. 모바일은 반응형으로 최소 커버.
- **월 지출 건수**: 지점당 약 60~80건/월. 12지점 합 720~960건/월.
- **연간 예산 변경 빈도**: 드물다 (경영계획 수립 시점 + 분기 검토)
- **Flex 연동**: 당분간 수동 (Flex 결재번호를 GM이 텍스트 입력)
- **슬랙 연동**: Phase 1은 없음. Phase 2에 검토.

### 9-3. 조직 가정

- **12개 지점 안정적**: Phase 1 기간 중 신규 지점 추가/폐점 없음 가정. 있으면 마스터 관리 UI로 대응.
- **카테고리 체계 안정**: 대카테고리 7개, 소카테고리 11개 당분간 유지. 변경 시 SGM/GM 승인.
- **GM 단독 예산 권한**: 추경/원 예산/마스터는 GM 1명에 집중. 부담 크면 Phase 2에 분산.

## 10. 성공 기준

### 10-1. Phase 1 MVP 완료 기준

**기능**
- [ ] 12개 지점 × 10카테고리 × 12월 예산 조회 가능
- [ ] 신도림 파일럿에서 1~4월 이관 데이터 집계가 시트 대시보드와 일치
- [ ] 월 90%/100% 경고 배지 정상 노출
- [ ] 미정 카테고리 등록 → SGM 알림 → 재분류 플로우 작동
- [ ] 추경 입력 → 실질 예산 자동 재계산
- [ ] 감사 로그에 모든 변경 기록

**운영**
- [ ] 신도림 파일럿 1주 성공 (지점 직원이 시트 없이 시스템만 사용 가능)
- [ ] 11개 지점 순차 온보딩 시작
- [ ] 5월 1일부터 신규 지출은 시스템에서만 등록

### 10-2. 장기 지향

- 지점 직원의 시스템 사용 시간 < 시트 사용 시간의 50% (효율 개선)
- 본사 PM의 "전사 현황 파악" 시간 < 기존의 30%
- 미정 카테고리 평균 재분류 시간 < 3일
- 연말 결산 시 시트 대시보드 합계 = 시스템 합계 (100% 정합)
